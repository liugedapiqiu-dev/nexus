# Desktop-Control / RGA — Feishu/Lark Verify-Only / Dry-Run 安全闸

> 项目线：desktop-control v2 / C 线
> 目标：在 **不真实盲发消息** 的前提下，为 Feishu / Lark 建立一条可重复执行的 verify-only / dry-run 桌面任务链。
> 验证结论应能证明：**能找对人、能确认输入框、能确认草稿、能在发送前停住。**
> 本文档是后续 live-run 前的最后安全闸，不是 live-run 放行令。

---

## 1. 文档定位

本文件基于以下已有基线继续细化：

- `acceptance/desktop-control-rga/ACCEPTANCE_BASELINE.md`
- `acceptance/desktop-control-rga/demo_safe_run.sh`

其中 baseline 已定义：

- 默认 `dry-run`；
- 高风险消息发送禁止盲发；
- 未完成目标确认门、发送前验证、人工确认，不得进入真实发送。

本文件进一步把上述原则压实到 **Feishu / Lark 聊天窗口** 场景，形成一条专用 verify-only 链路。

---

## 2. 本链路的结论边界

本链路 **只能证明**：

1. 能定位到候选 Feishu/Lark 会话；
2. 能确认当前可见对象是预期联系人/群聊；
3. 能确认焦点在消息输入框；
4. 能把草稿放入输入框或粘贴到输入框；
5. 能在真实发送动作前强制停住；
6. 若出现歧义、焦点漂移、输入法异常、视觉锚点不稳，能中止并交还人工。

本链路 **不能证明**：

1. 已具备真实外发授权；
2. 已确认 Enter / Cmd+Enter / 点击发送在当前 Feishu/Lark 配置下的真实语义；
3. 可以无人监督自动向真实联系人发送；
4. dry-run 结果可直接等价为 live-run 通过。

---

## 3. 适用范围

适用于：

- macOS 桌面 Feishu / Lark 客户端；
- 聊天对象包含单聊、内部群、测试群；
- 以 desktop-control / RGA 执行 GUI 定位、激活、截图、输入、停前验证。

不适用于：

- API 直发；
- 浏览器版 Feishu/Lark 的无差别迁移；
- 批量联系人；
- 默认真实外发。

---

## 4. 运行级别定义

### 4.1 Verify-Only（默认）

这是本文件的主模式。

允许：

- 激活 Feishu/Lark；
- 搜索并进入候选联系人/群聊；
- 识别标题、头像、会话名称、群名等锚点；
- 确认输入框存在；
- 输入或粘贴草稿；
- 在发送前停住；
- 截图留证；
- 要求人工确认。

禁止：

- 按下可能触发发送的最终快捷键；
- 点击真实“发送”按钮；
- 对真实对象盲发；
- 自动重试发送；
- 一次跑多个会话。

### 4.2 Live-Run（不在本文默认范围内）

只有在本文 verify-only 完整通过、且另有单独人工放行时，才允许设计 live-run。

本文本身 **不授予** live-run 资格。

---

## 5. Feishu/Lark 场景下的最小确认链

针对 Feishu/Lark，最小确认链固定为 4 段：

1. **找对人 / 找对群**
2. **找对输入框**
3. **找对草稿内容**
4. **在发送前停住**

只要有任一段无法可靠确认，就判定为：

- `FAIL`，或
- `MANUAL`

且不得继续推进。

---

## 6. Feishu/Lark Verify-Only 标准流程

## 6.1 Step A — 应用激活

目标：确认前台确实是 Feishu/Lark 客户端，而不是其他 IM、浏览器、系统弹窗或错误窗口。

### 自动可确认

- 应用名称包含 `Feishu` / `Lark`；
- 主窗口已置前；
- 没有明显权限弹窗或系统级遮挡；
- 当前截图可见 Feishu/Lark 典型侧边栏 / 顶栏结构。

### 仍需人工确认

- 当前看到的窗口确实是本次要操作的会话工作区，而不是设置页、文件页、通知页。

### 通过标准

- 前台应用识别成功；
- 截图中可见可用的聊天界面结构；
- 若识别不到聊天结构，则停止。

---

## 6.2 Step B — 目标会话确认（找对人）

目标：证明当前候选会话确实是预期联系人 / 群聊，而不是同名、近似名、历史会话或搜索结果偏移。

### 推荐锚点

至少采集以下锚点中的 3 项：

1. 顶部会话标题；
2. 左侧会话列表高亮项；
3. 头像 / 群头像形态；
4. 群聊成员标签 / 备注信息；
5. 搜索结果选中后的标题一致性；
6. 最近消息上下文与目标身份一致。

### 自动可确认

- 顶部标题文本与目标名精确匹配或高置信匹配；
- 左侧高亮项与顶部标题一致；
- 视觉上能定位唯一当前会话。

### 仍需人工确认

以下任一情况必须人工确认：

- 同名联系人多个；
- 群名高度相似；
- 备注名和真实姓名冲突；
- 标题识别稳定性不足；
- 截图中无法清楚区分是单聊还是群聊。

### 通过标准

- 自动锚点 ≥ 2 个成立；
- 且人工明确确认“就是这个对象”；
- 否则不得进入输入框确认。

---

## 6.3 Step C — 输入框确认（找对输入框）

目标：证明焦点落在消息输入框，而不是搜索框、顶部查找框、侧边过滤框、文件标题栏、浏览器地址栏等。

### 自动可确认

- 可见消息输入区域位于聊天主区底部；
- 输入框边界、占位提示、工具栏关系符合聊天输入区结构；
- 点击后出现文本光标或高亮态；
- 输入框内容变化仅发生在底部消息编辑区。

### 风险误区

Feishu/Lark 中容易误判的区域包括：

- 左侧搜索框；
- 聊天内“搜索消息”入口；
- 文档/文件重命名输入框；
- 日程或任务标题输入框；
- 系统级聚焦漂移到别的应用。

### 仍需人工确认

- 当底部编辑区样式被主题、缩放、插件或弹窗改变时；
- 当无法稳定看见文本光标；
- 当截图不足以证明输入位置时。

### 通过标准

- 底部聊天输入区被识别；
- 焦点进入后仅底部编辑区发生变化；
- 不允许“看起来像输入框”就继续。

---

## 6.4 Step D — 草稿确认（找对草稿）

目标：证明输入框中展示的草稿就是计划文本，而不是截断、乱码、拼音残留、半上屏状态或旧草稿混入。

### 推荐写入策略

优先级从高到低：

1. 剪贴板粘贴短文本；
2. 受控字符级输入；
3. 若中文输入法不稳定，回退为仅验证剪贴板内容，不进入 live-run。

### 自动可确认

- 输入框可见文本与计划文本一致；
- 没有多余前后缀；
- 没有明显拼音候选框残留；
- 没有旧草稿残留未清理；
- 若为多行文本，多行结构可见。

### 仍需人工确认

- 文本过长导致输入框折叠；
- 仅能看到前半部分；
- 中文上屏不稳定；
- markdown / emoji / @ 提及语义无法纯视觉稳定判断。

### 通过标准

- 草稿可见部分与计划内容一致；
- 人工确认无误发语义风险；
- 若看不清全文，则仅可判 `MANUAL`，不得默认通过。

---

## 6.5 Step E — 发送前停住（最后安全闸）

目标：证明系统能在“已经找对对象 + 找到输入框 + 放入草稿”之后，**仍然不自动真实外发**。

### 强制规则

- 默认禁止按 Enter；
- 默认禁止点击发送按钮；
- 默认禁止组合键发送；
- 默认禁止任何“再确认后自动发送”的隐式行为；
- 没有人工新指令，动作必须停在“草稿可见、未发出”。

### 自动可确认

- 输入框中草稿仍然可见；
- 消息气泡未新增；
- 发送按钮未被触发；
- 会话中未出现新的自己发出的消息记录。

### 仍需人工确认

- 肉眼确认“当前确实还没发出去”；
- 若 UI 延迟或消息回显不明显，人工必须参与判定。

### 通过标准

- 运行结束时，会话保持在“待发送草稿”状态；
- 没有新发送气泡；
- 没有任何外发证据。

---

## 7. 当前可自动确认 vs 仍需人工确认

## 7.1 当前已可自动确认的环节

在桌面控制 / RGA + 截图判定能力足够稳定时，可自动确认：

1. Feishu/Lark 应用是否在前台；
2. 主聊天界面是否出现；
3. 候选会话标题是否匹配；
4. 会话列表高亮与顶部标题是否一致；
5. 底部消息输入区是否存在；
6. 输入后底部草稿是否出现；
7. 运行结束时是否仍停在未发送状态。

## 7.2 当前仍需人工确认的环节

以下环节必须保留人工：

1. 最终确认“这个联系人/群聊就是这次目标”；
2. 最终确认草稿语义正确、无歧义；
3. 最终确认当前模式仍是 verify-only / dry-run；
4. 最终确认未发出；
5. 若要从 verify-only 升级到 live-run，必须人工重新明确授权。

## 7.3 自动化的边界意见

我的判断是：

- “找对人”可以做高置信自动辅助，但不应取消人工最后确认；
- “找对输入框”是可以较稳定自动确认的；
- “草稿对不对”可自动比对可见文本，但语义正确性仍要人工兜底；
- “发送前停住”必须是默认强约束，而不是依赖模型临场自觉。

---

## 8. 通过标准（Feishu/Lark 专项）

本专项链路判定 `PASS` 必须满足以下全部条件：

1. 前台应用成功锁定为 Feishu/Lark；
2. 目标会话被识别，且人工确认无误；
3. 消息输入框被确认；
4. 草稿内容被放入并可见确认；
5. 运行在发送前停住；
6. 没有任何真实新消息外发证据；
7. 任一异常时默认中止，而不是继续尝试发送。

可判 `PASS-WITH-GAPS` 的情况：

- 可完成停前验证，但长文本仅能部分可见；
- 会话定位需要较多人工辅助；
- 输入法稳定性不足，仅完成短文本验证。

判 `FAIL` 的情况：

- 目标对象无法可靠确认；
- 输入框识别不稳；
- 草稿内容无法核对；
- 运行过程中发生疑似真实发送；
- 失败后仍继续推进。

判 `MANUAL` 的情况：

- 截图证据不足；
- UI 主题/缩放/弹窗导致无法稳定识别；
- 需要人工继续处理，但没有明显错误。

---

## 9. 禁止事项

本专项明确禁止：

1. 把真实客户、供应商、外部陌生联系人作为 verify-only 首测对象；
2. 用 Enter 试探 Feishu/Lark 是否会发送；
3. 在 verify-only 中点击真实发送按钮；
4. 在目标模糊时继续输入草稿；
5. 因识别失败而自动重试多个聊天窗口；
6. 把“草稿已出现”误当作“已可 live-run”；
7. 没有人类在环时真实外发；
8. 把这份文档当作 live-run 批准书。

---

## 10. 推荐的实际验收执行顺序

建议按以下顺序落地：

1. 先用 **自己的测试号 / 自测群 / 空白内部测试群**；
2. 跑 verify-only；
3. 留存 3 类证据：
   - 会话确认截图；
   - 输入框 + 草稿截图；
   - 结束时“未发送”截图；
4. 用本文件判定 PASS / PASS-WITH-GAPS / FAIL / MANUAL；
5. 只有专项 PASS 后，才允许单独起草 live-run 方案。

## 11. 与 live-run 的衔接规则

若未来进入 live-run，必须额外补齐：

1. 目标测试对象白名单；
2. Feishu/Lark 当前发送语义确认（Enter / Cmd+Enter / 点击发送）；
3. 单独的 live-run 授权记录；
4. 发送后验证与补救预案；
5. “只发一次、失败不补发”的执行约束。

在这些条件未补齐前，本文档的结论只能是：

> **Feishu/Lark verify-only / dry-run 链路可作为 live-run 前的重要安全闸，但不是最终放行本身。**

若要进入更接近 live-run 的最终放行阶段，请继续使用：

- `acceptance/desktop-control-rga/FEISHU_LARK_LIVE_RUN_FINAL_GATE.md`
- `acceptance/desktop-control-rga/feishu_lark_live_run_ready_gate.sh`

其中 final gate 的目标是把结论推进到：

- `READY-FOR-HUMAN-AUTH`（可进入人工最终授权前状态）

而不是：

- 自动真实外发
- 默认开启 live-run
- 无人监督发送

---

## 12. 本次交付结论

基于现有 baseline，可以正式确认：

- desktop-control / RGA 在 Feishu/Lark 上的正确目标不是“自动发出去”，而是“自动停在发送前”；
- verify-only 的核心验收不是成功外发，而是成功证明：
  - 找对人；
  - 找对输入框；
  - 找对草稿；
  - 在发送前停住；
- 后续若要 live-run，必须另立更严格放行条件。
